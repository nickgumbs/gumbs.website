# Repository Guidelines

## Project Structure & Module Organization
- `src/` contains the JavaScript entrypoint (`src/index.js`), Handlebars templates (`src/templates/`), helpers (`src/helpers/`), and styles (`src/styles.css`).
- `public/` holds static assets and content, including `public/index.html` and `public/data/homepage.json`.
- `cypress/` and `playwright/` contain end-to-end tests and page objects.
- `ops/` houses Terraform infrastructure modules and the `ops/prod/` environment.
- `dist/` is generated by Webpack builds and should not be edited by hand.

## Ops 
- We are in the process of migrating existing Terraform deployment into re-usable modules to create multiple environments.
- `prod/` should contain infrastructure for my production site at `https://gumbs.me`
- `staging/` should contain infrastructure configuration for my test site at `https://staging.gumbs.me`

## Build, Test, and Development Commands
- `npm run webpack:start` starts the Webpack dev server on `http://localhost:8080` with hot reload.
- `npm run webpack:build` builds a production bundle into `dist/`.
- `npm run cypress:open` launches Cypress in interactive mode; `npm run cypress:run` runs headless.
- `npm run playwright:run` runs Playwright tests from `playwright/e2e` (defaults to the production base URL).
- `npm run prettier:format` formats JS/CSS/JSON with the repo Prettier config.

## Coding Style & Naming Conventions
- Formatting is enforced with Prettier (`.prettierrc`): 2-space indentation, single quotes, semicolons, 80-char print width.
- Follow existing file naming: `Header.hbs`/`Footer.hbs` for templates, camelCase for JS helpers (e.g., `handlebars.js`), and kebab-case for asset folders.
- Keep Handlebars partials under `src/templates/partials/` and register helpers in `src/helpers/handlebars.js`.

## Testing Guidelines
- Cypress specs live in `cypress/e2e` and use the `.cy.js` suffix; run against the local dev server.
- Playwright specs live in `playwright/e2e` and use the `.pw.js` suffix; base URL is `https://gumbs.me` in config.
- There is no explicit coverage gate; prioritize UI regressions, accessibility, and critical flows.
- After making any changes, run cypress and playwright test suites.
- When updating UI components and templates, add new Cypress or Playwright UI test to validate.

## Commit & Pull Request Guidelines
- Use short, descriptive, sentence-style commit subjects (e.g., “Complete Migration for Main Site”).
- PRs should include a clear summary, testing notes (commands + results), and screenshots for UI changes.
- For Terraform changes under `ops/`, note affected resources and any required manual steps.

## Configuration & Secrets
- Keep secrets out of the repo; use environment variables or local tooling when needed.
- Infrastructure changes require valid AWS credentials in your local environment.
